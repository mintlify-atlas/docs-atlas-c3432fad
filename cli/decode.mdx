## Synopsis

```bash
sef decode [OPTIONS]
```

## Description

Reads droplet files from the specified input directory, runs the error-resilient peeling decoder on each epoch, and writes recovered blocks to the output directory in Bitcoin's standard `blk*.dat` format.

The decoder:
1. Scans for `epoch_N/` subdirectories
2. Loads droplets and manifest metadata for each epoch
3. Runs the peeling algorithm with hash-based verification
4. Reconstructs symbols back into blocks
5. Writes blocks to `blk*.dat` files with network magic bytes

<Note>
  The decoder requires `manifest.bin` files in each epoch directory. These are automatically generated by the [`generate`](/cli/generate) command when symbol normalization is enabled.
</Note>

## Options

<ParamField path="input" type="path" default="droplets">
  Path to the droplets directory containing `epoch_N/` subdirectories.
  
  **Short flag:** `-i`
</ParamField>

<ParamField path="output" type="path" default="decoded">
  Output directory for recovered block files.
  
  The command writes blocks in Bitcoin Core's `blk*.dat` format with network magic bytes and size prefixes.
  
  **Short flag:** `-o`
</ParamField>

<ParamField path="epoch" type="integer">
  Only decode this specific epoch index (optional).
  
  If not specified, all epochs in the input directory are decoded.
  
  **Example:** `--epoch 42`
</ParamField>

## Output Format

### blk*.dat Files

Recovered blocks are written to `blk*.dat` files following Bitcoin Core's format:

```
<output>/
├── blk00000.dat    # First ~128MB of blocks
├── blk00001.dat    # Next ~128MB
└── ...
```

Each file contains concatenated blocks with the structure:

```
[magic bytes (4)]  [block size (4)]  [block data (N)]
[magic bytes (4)]  [block size (4)]  [block data (N)]
...
```

- **Magic bytes:** Signet network magic `0x0a03cf40`
- **Block size:** Little-endian uint32
- **Block data:** Raw consensus-serialized block

Files are rotated at 128 MiB to match Bitcoin Core's file size limits.

## Decoding Algorithm

The decoder uses the **error-resilient peeling algorithm** from the SeF paper:

1. **Initialize:** Load all droplets and create adjacency graph
2. **Peel:** Find droplets of degree 1 (connected to single symbol)
3. **Verify:** Check symbol hash against manifest
4. **Propagate:** XOR recovered symbol from all connected droplets
5. **Repeat:** Until all symbols are recovered or decoding stalls

### Verification

Each recovered symbol is verified against its hash in the manifest before propagation. This prevents murky (adversarial) droplets from corrupting the decoding process.

<Note>
  **Decoding stall:** Occurs when no degree-1 droplets remain but some symbols are unrecovered. This indicates insufficient droplets or too many corrupted droplets.
</Note>

## Recovery Guarantees

With parameters `k`, `c`, `delta` and `n` droplets:
- **Success probability:** ≥ 1 - δ when `n ≥ k + O(√k · ln²(k/δ))`
- **Typical overhead:** 2× (`n = 2k`) provides >99% success for `k=100`, `δ=0.05`

### Failure Modes

| Failure Type | Cause | Solution |
|--------------|-------|----------|
| Decoding stall | Insufficient droplets | Increase `n` during generation |
| Verify failures | Corrupted droplets | Use authenticated droplet transport |
| Missing manifest | Symbol normalization disabled | Enable `--symbol-size` during generation |

## Performance

### Decoding Throughput

Typical throughput on modern hardware:
- **Small epochs (k≤100):** 10–50 MB/s
- **Large epochs (k≥1000):** 50–200 MB/s

Decoding is faster than encoding due to the incremental peeling process.

### Memory Usage

Memory requirements:
- **Per epoch:** O(n × average_droplet_size)
- **Typical:** 10–100 MB per epoch for `k=100`, `n=200`

<Tip>
  Use `--epoch` to decode one epoch at a time if memory is constrained.
</Tip>

## Troubleshooting

### "No manifest.bin" Error

**Cause:** Epoch was generated without symbol normalization.

**Solution:** Regenerate droplets with `--symbol-size > 0`.

### Low Recovery Rate

**Cause:** Insufficient droplets per epoch.

**Solution:** Increase `--n` during generation or decrease `k` (smaller epochs).

### Verify Failures

**Cause:** Droplet file corruption or hash mismatch.

**Solution:** Verify droplet file integrity and regenerate if necessary.
