## Synopsis

```bash
sef experiment [OPTIONS]
```

## Description

Runs two types of experiments from the SeF paper:

1. **Decoding Threshold Sweep:** Measures success rate vs. number of droplets
2. **Multi-Node Storage Reduction:** Simulates distributed droplet nodes with storage factor γ = k/s

These experiments use **graph-only simulations** (no actual XOR payloads) to quickly measure decoding success rates across parameter configurations.

<Note>
  This command runs simulations only. It does not encode/decode actual blockchain data. Use [`reconstruct`](/cli/reconstruct) for end-to-end testing with real blocks.
</Note>

## Options

<ParamField path="k" type="integer" default="100">
  Epoch size — number of source blocks/symbols per epoch.
  
  **Short flag:** `-k`
</ParamField>

<ParamField path="c" type="float" default="0.1">
  Robust Soliton parameter c — tuning constant.
  
  **Short flag:** `-c`
</ParamField>

<ParamField path="delta" type="float" default="0.05">
  Robust Soliton parameter delta — failure probability.
</ParamField>

<ParamField path="pool-size" type="integer" default="500">
  Total droplet pool size for multi-node experiments.
  
  This is the maximum number of droplets available across all nodes.
</ParamField>

<ParamField path="trials" type="integer" default="500">
  Number of trials per configuration.
  
  Higher values provide more accurate statistics but take longer to run.
  
  **Short flag:** `-t`
</ParamField>

## Understanding the Output

### Part 1: Decoding Threshold Sweep

Shows success rate as a function of total droplets M:

| Column | Description |
|--------|-------------|
| `M` | Total number of droplets available for decoding |
| `M/k` | Overhead ratio (M/k = 1.0 means no overhead) |
| `success%` | Percentage of trials where all k symbols were recovered |
| `avg_decoded` | Average number of symbols decoded across all trials |

**Key metrics:**
- **M* for 95%:** Minimum droplets needed for 95% success rate
- **M* for 99%:** Minimum droplets needed for 99% success rate

<Note>
  For k=100 with default parameters, M* ≈ 1.2k provides 95% success. This matches the theoretical bound k + O(√k·ln²(k/δ)).
</Note>

### Part 2: Multi-Node Storage Reduction

Simulates distributed storage where each of K nodes stores s droplets:

| Column | Description |
|--------|-------------|
| `K` | Number of nodes contacted |
| `avg_M` | Average number of distinct droplets obtained (accounting for duplicates) |
| `success%` | Percentage of trials where all k symbols were recovered |
| `avg_decoded` | Average number of symbols decoded |

**Key metrics:**
- **γ (gamma):** Storage reduction factor = k/s
  - γ=20x means each node stores 1/20th of the data
- **K for 95%:** Number of nodes needed for 95% success

## Storage Reduction Analysis

### How γ Affects Requirements

For k=100 with c=0.1, δ=0.05:

| γ | s (per node) | K for 95% | Total contacted | Storage per node |
|---|--------------|-----------|------------------|------------------|
| 20x | 5 | 19 | 95 droplets | 5% of k |
| 10x | 10 | 8 | 75 droplets | 10% of k |
| 5x | 20 | 4 | 76 droplets | 20% of k |
| 2x | 50 | 2 | 95 droplets | 50% of k |

**Interpretation:**
- **Higher γ:** Better storage savings, but more nodes needed
- **Lower γ:** Fewer nodes needed, but less storage savings

The SeF paper proves that K = O(√k·ln²(k/δ)) honest nodes suffice for γ=k/s.

### Duplicate Droplets

Notice that `avg_M` < K×s due to duplicates:

```
# γ=20x, s=5, K=15
avg_M = 70.2 < 15×5 = 75
```

Duplicates reduce effectiveness. In practice, nodes should coordinate to minimize overlap.

## Experiment Methodology

### Graph-Only Simulation

The experiment uses **peeling check** (graph-based) rather than full encoding:

1. Sample M droplet degree sequences from RSD
2. Build bipartite graph (droplets → symbols)
3. Run peeling decoder on graph structure
4. Record whether all k symbols are reached

**Advantages:**
- 100–1000× faster than full XOR encoding
- Sufficient for measuring decoding success rates
- Exact results for fountain code graph properties

**Limitations:**
- Does not measure XOR throughput
- Does not test hash verification
- Assumes perfect droplet transmission

<Tip>
  Use `experiment` for quick parameter sweeps. Use [`reconstruct`](/cli/reconstruct) for real-world validation with blockchain data.
</Tip>

## Use Cases

### Finding Optimal Overhead

Determine minimum M/k ratio for target success rate:

```bash
sef experiment --k 100 --trials 1000
# Output: M* for 99% = 130 (1.30x overhead)

# Use this in generate:
sef generate --blocks-dir ~/.bitcoin/signet/blocks --k 100 --n 130
```

### Comparing Parameters

Test different c and δ values:

```bash
# Baseline
sef experiment --k 100 --c 0.1 --delta 0.05 --trials 500
# M* for 95%: 120

# Lower c (more robust)
sef experiment --k 100 --c 0.05 --delta 0.05 --trials 500
# M* for 95%: 115 (better, but higher E[d])

# Higher delta (less robust)
sef experiment --k 100 --c 0.1 --delta 0.1 --trials 500
# M* for 95%: 125 (worse)
```

### Multi-Node Network Design

Plan distributed storage network:

```bash
sef experiment --k 1000 --trials 500
# γ=20x: K for 95% = 180 nodes
# γ=10x: K for 95% = 85 nodes

# Choose γ=10x for better reliability/cost trade-off
```

## Performance

Typical runtime on modern hardware:

| Configuration | Trials | Time |
|---------------|--------|------|
| k=100, t=500 | 500 | 5–15s |
| k=100, t=1000 | 1000 | 10–30s |
| k=500, t=500 | 500 | 30–60s |
| k=1000, t=500 | 500 | 60–120s |

Runtime scales as O(k² × trials) due to peeling decoder complexity.

  Use `--trials 100` for quick tests during development. Use `--trials 1000` for publishable results.

## Interpreting Results

### Threshold Behavior

Notice the sharp transition in success rate:

```
M=110: 45.7% success
M=120: 94.2% success  ← threshold
M=130: 99.8% success
```

This is the **fountain code threshold phenomenon**: below threshold, decoding almost always fails; above threshold, decoding almost always succeeds.

### Variance

With 500 trials:
- **95% confidence interval:** ±1.4% on success rate
- **99% confidence interval:** ±1.8% on success rate

Increase `--trials` if you need tighter bounds.

## Limitations

### No Adversarial Droplets

The experiment assumes all droplets are honest. In practice:
- Some nodes may provide murky (adversarial) droplets
- The decoder detects these via hash verification
- You may need more than M* droplets if some are murky

### No Network Effects

The simulation assumes:
- Perfect droplet transmission
- No latency or bandwidth constraints
- Uniform node availability

Real networks have additional overhead.

### No XOR Cost

Graph-only simulation doesn't measure:
- Encoding/decoding throughput (MB/s)
- Memory usage
- Cache effects

Use [`reconstruct`](/cli/reconstruct) for performance benchmarks.
