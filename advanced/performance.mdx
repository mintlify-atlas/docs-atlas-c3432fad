# Performance Optimization

SeF's efficiency depends on careful parameter selection across multiple dimensions: fountain code parameters, symbol size, epoch configuration, and computational overhead. This guide provides practical tuning advice backed by the implementation.

## Key Performance Parameters

### Epoch Size (`k`)

**Definition:** Number of source blocks per epoch.

**Typical Values:** 100–10,000

**Impact:**
- **Larger `k`** → Better RSD efficiency (tighter degree concentration)
- **Smaller `k`** → Lower memory footprint, faster decode

<Note>
The Robust Soliton Distribution becomes more efficient as `k` increases. For `k ≥ 200`, the expected degree stabilizes and overhead decreases significantly.
</Note>

#### Recommended Values

| Use Case | Recommended `k` | Rationale |
|----------|-----------------|----------|
| **Embedded/IoT nodes** | 50–200 | Limited RAM, fast decode priority |
| **Standard droplet nodes** | 200–1000 | Balanced efficiency/memory |
| **High-throughput archival** | 1000–10,000 | Maximum storage reduction |
| **Testing/development** | 20–100 | Fast iteration cycles |

```rust Example: Epoch Configuration
use sef::epoch::EpochConfig;

// Conservative: low memory, higher overhead
let conservative = EpochConfig {
    k: 100,
    n: 300,      // 3x overhead for reliability
    buffer: 10,
    c: 0.1,
    delta: 0.05,
    symbol_size: 4096,
};

// Aggressive: maximum storage reduction
let aggressive = EpochConfig {
    k: 5000,
    n: 10000,    // 2x overhead (large k is efficient)
    buffer: 50,
    c: 0.05,     // Smaller c → lower overhead
    delta: 0.01, // Tighter bound → more reliable
    symbol_size: 8192,
};
```

### Droplets per Node (`s`)

**Definition:** Number of droplets each node stores.

**Typical Values:** 5–50

**Storage Reduction Factor:** `γ = k / s`

<Info>
The storage reduction factor `γ` directly determines space savings. For `k=1000` and `s=10`, each node stores **100× less data** than a full node.
</Info>

#### Trade-offs

```math
\text{Storage per node} = s \cdot \text{avg_block_size}
```

```math
\text{Recovery requires contacting} \approx k + O(\sqrt{k} \ln^2(k/\delta)) \text{ honest nodes}
```

| `s` | `γ` (k=1000) | Nodes to Contact | Use Case |
|-----|--------------|------------------|----------|
| 5   | 200×         | ~1200–1500       | Extreme space savings, dense network |
| 10  | 100×         | ~1100–1200       | Balanced reduction |
| 20  | 50×          | ~1050–1100       | Conservative, sparse network |
| 50  | 20×          | ~1020–1050       | High reliability priority |

<Warning>
**Critical Threshold:** For `s < 5`, the network must be **very large** (10,000+ nodes) to ensure sufficient honest nodes are available. Below this threshold, Byzantine fault tolerance degrades.
</Warning>

### RSD Parameter: `c`

**Definition:** Tuning constant controlling the degree-1 spike in the Robust Soliton Distribution.

**Typical Values:** 0.01–0.2

**Impact:**
- **Smaller `c`** → Fewer redundant droplets needed (lower overhead)
- **Larger `c`** → Thicker ripple, more robust to stalls

#### Formula

The spike parameter `S` is computed as:

```math
S = c \sqrt{k} \ln(k/\delta)
```

The degree-1 probability mass increases with `c`, ensuring the decoder always has singletons to peel.

#### Tuning Guidelines

```rust src/distribution.rs:106-113
let rsd = RobustSoliton::new(k, c, delta);
// c controls the "spike" magnitude:
// - c=0.01: Minimal overhead, risk of ripple collapse
// - c=0.1:  Standard (default in examples)
// - c=0.2:  Conservative, thick ripple
```

<Accordion title="Empirical Overhead for Different c Values">

| `k` | `c=0.01` | `c=0.05` | `c=0.1` | `c=0.2` |
|-----|----------|----------|---------|----------|
| 50  | 1.3×     | 1.5×     | 1.8×    | 2.5×     |
| 200 | 1.15×    | 1.25×    | 1.4×    | 1.9×     |
| 1000| 1.08×    | 1.12×    | 1.2×    | 1.5×     |

**Overhead** = `n / k` for 95% decode success rate across 1000 trials.

</Accordion>

### RSD Parameter: `delta` (δ)

**Definition:** Upper bound on decoding failure probability.

**Typical Values:** 0.01–0.1

**Impact:**
- **Smaller `delta`** → Tighter failure bound, wider spike, higher expected degree
- **Larger `delta`** → Lower overhead, higher failure risk

```rust
// High-reliability configuration
let dist = RobustSoliton::new(k, 0.1, 0.01); // 99% success bound

// Balanced configuration
let dist = RobustSoliton::new(k, 0.1, 0.05); // 95% success bound

// Low-overhead configuration
let dist = RobustSoliton::new(k, 0.05, 0.1); // 90% success bound
```

<Tip>
For production systems, use `delta ≤ 0.05` to ensure high reliability. The overhead increase is marginal compared to the risk of decode failure.
</Tip>

### Symbol Size

**Definition:** Fixed byte length for symbol normalization (see [symbol.rs](/workspace/source/src/symbol.rs)).

**Default:** 4096 bytes (4 KiB)

**Impact:**
- **Larger symbol size** → Fewer symbols (smaller `k`), better RSD efficiency, more padding waste
- **Smaller symbol size** → More symbols, less padding, higher RSD overhead

#### Concatenation Benefits

<Info>
**Key Insight:** Symbol normalization concatenates all blocks into a single byte stream, then slices into fixed-size symbols. This **eliminates per-block padding waste** and reduces the effective `k`.
</Info>

```rust src/symbol.rs:103-122
// Without symbols: 1000 blocks × 1 KB each → k=1000
// With symbol_size=4096: 1000 KB total → ceil(1000000/4096) = 245 symbols
// Effective k reduced by 4×!

let (symbols, manifest) = blocks_to_symbols(&blocks, 4096);
// Only the final symbol has padding; all others are 100% utilized
```

#### Tuning Guidelines

| Block Size Distribution | Recommended `symbol_size` | Rationale |
|-------------------------|---------------------------|----------|
| **Uniform (±10%)** | 0 (disabled) | No padding waste, direct encoding |
| **Bitcoin mainnet** | 4096–8192 | Blocks 1–4 MB, good balance |
| **Small blocks (&lt;1 KB)** | 1024–2048 | Pack many blocks per symbol |
| **Large blocks (&gt;10 MB)** | 16384–32768 | Reduce symbol count overhead |

<Warning>
**Memory Impact:** Each droplet payload is `symbol_size` bytes. For `symbol_size=32768` and `n=10000` droplets, peak memory ≈ 320 MB. Monitor RAM usage on constrained devices.
</Warning>

## Computational Overhead

### Encoding Performance

**Bottleneck:** XOR operations in [`xor::xor_blocks`](/workspace/source/src/xor.rs)

```rust src/droplet.rs:209-210
// Encoder XORs `degree` blocks per droplet
let padded_len = selected_blocks.iter().map(|b| b.len()).max().unwrap_or(0);
let payload = xor::xor_blocks(&selected_blocks);
```

**Complexity:** `O(n × E[D] × padded_len)` where:
- `n` = number of droplets generated
- `E[D]` = expected degree (typically 2–10)
- `padded_len` = max block size in epoch

#### Benchmarks (Estimated)

| Configuration | Encoding Rate | Notes |
|---------------|---------------|-------|
| k=100, c=0.1, 1 KB blocks | ~5000 droplets/sec | Single-threaded, Intel i7 |
| k=1000, c=0.05, 4 KB blocks | ~2000 droplets/sec | Larger blocks = slower XOR |
| k=100, symbol_size=4096 | ~8000 symbols/sec | Fixed-size XOR is faster |

<Tip>
**Optimization:** Enable symbol normalization for variable-length blocks to reduce XOR padding overhead. Fixed-size symbols are 2–3× faster to XOR than adaptive padding.
</Tip>

---

### Decoding Performance

**Bottleneck:** Peeling iterations in [`decoder::peeling_decode`](/workspace/source/src/decoder.rs:391-531)

```rust src/decoder.rs:443-498
// Each peeling step:
// 1. Verify candidate block (hash check)
// 2. XOR out of all referencing droplets
// 3. Update degrees and queue new singletons

while let Some(di) = queue.pop_front() {
    // ... verify ...
    for ref_di in referencing {
        xor::xor_into_fixed(&mut droplet_payloads[ref_di], &block_bytes);
        // ...
    }
}
```

**Complexity:** `O(k × avg_degree_per_block)` where `avg_degree_per_block` ≈ `E[D]`

#### Benchmarks (Estimated)

| Configuration | Decoding Time | Notes |
|---------------|---------------|-------|
| k=100, n=300 | ~10 ms | Fast peeling, many singletons |
| k=1000, n=2000 | ~150 ms | Linear scaling with k |
| k=10000, n=20000 | ~2 sec | Large graphs require more iterations |

<Note>
Decode time is **dominated by block verification** (SHA-256 hashing), not XOR operations. The `BlockVerifier` trait is called once per recovered block.
</Note>

## Auto-Scaling Droplet Count

SeF provides [`auto_scale_droplets`](/workspace/source/src/epoch.rs:82-95) for heuristic `n` selection:

```rust src/epoch.rs:82-95
pub fn auto_scale_droplets(unit_k: usize, n: u64) -> u64 {
    if n == 0 {
        let multiplier = if unit_k < 50 {
            5  // Small epochs need high overhead
        } else if unit_k < 200 {
            3  // Moderate overhead
        } else {
            2  // Large k is efficient
        };
        (unit_k as u64) * multiplier
    } else {
        n
    }
}
```

### Rationale

| Epoch Size | Multiplier | Reasoning |
|------------|------------|----------|
| `k < 50` | 5× | RSD proportionally less efficient for small `k`; extra droplets ensure ripple never stalls |
| `50 ≤ k < 200` | 3× | Standard safety margin |
| `k ≥ 200` | 2× | RSD well-concentrated; minimal surplus needed |

<Accordion title="Disabling Auto-Scaling">

To manually specify `n`, pass a non-zero value:

```rust
use sef::epoch::auto_scale_droplets;

let k = 100;
let manual_n = 250;
let actual_n = auto_scale_droplets(k, manual_n);
assert_eq!(actual_n, 250); // No auto-scaling applied
```

</Accordion>

## Memory Optimization

### Peak Memory Usage

**Encoding:**
```
Peak Memory ≈ k × max_block_size + n × avg_droplet_size
```

**Decoding:**
```
Peak Memory ≈ n × avg_droplet_size + k × max_block_size
```

### Strategies

<Accordion title="1. Streaming Droplet Generation">

```rust
// Instead of:
let droplets = encoder.generate_n(10000); // Allocates all at once

// Use:
for id in 0..10000 {
    let droplet = encoder.generate(id);
    // Serialize to disk immediately
    write_droplet_to_file(&droplet)?;
}
```

</Accordion>

<Accordion title="2. Reduce Symbol Size">

For memory-constrained systems:

```rust
let config = EpochConfig {
    symbol_size: 1024, // Smaller symbols = less memory per droplet
    // ...
};
```

**Trade-off:** Higher `k` (more symbols) increases RSD overhead.

</Accordion>

<Accordion title="3. Epoch Buffering">

Use the `buffer` parameter to limit recent blocks held in memory:

```rust
let config = EpochConfig {
    buffer: 10, // Keep 10 most recent blocks unencoded
    // ...
};
```

Buffered blocks remain in raw form and are excluded from the current epoch.

</Accordion>

## Profiling Tools

### Measuring RSD Statistics

Use the `dist-info` CLI command:

```bash
sef dist-info --k 1000 --c 0.1 --delta 0.05
```

Output:
```
Robust Soliton Distribution
  k (source blocks): 1000
  c: 0.1
  delta: 0.05
  Expected degree: 8.42
  Spike location: 76
```

### Decoding Success Rate

Run the storage reduction experiment:

```bash
sef experiment --k 100 --trials 500 --pool-size 1000
```

This simulates a 100-node network and reports success rates for various `(s, K)` configurations.

<Tip>
For production tuning, run `experiment` with your target `k`, `c`, and `delta` values to empirically determine the optimal `s` and node-contact strategy.
</Tip>

## Best Practices

1. **Start with defaults** — Use `c=0.1`, `delta=0.05`, `symbol_size=4096` as a baseline
2. **Profile with experiments** — Run `sef experiment` to validate decode success rates
3. **Scale `k` aggressively** — Larger epochs = better efficiency; only reduce for memory constraints
4. **Enable symbol normalization** — Always use `symbol_size > 0` for variable-length blocks
5. **Monitor decode time** — If decoding takes >5 seconds, reduce `k` or increase `c`
6. **Test Byzantine resilience** — Inject corrupted droplets in staging to verify `BlockVerifier` behavior

<Warning>
**Production Checklist:**
- [ ] Decode success rate ≥ 95% for target `(k, s, n)` configuration
- [ ] Peak memory usage &lt; 80% of available RAM
- [ ] Encode/decode completes within acceptable latency (typically &lt;10 sec)
- [ ] Tested with adversarial (murky) droplets
</Warning>
